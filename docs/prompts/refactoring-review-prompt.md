# 리팩토링 검토 프롬프트

아래 코드베이스 또는 지정된 모듈/디렉터리를 리팩토링 관점에서만 검토해줘.

이 작업은 검토 작업이다.
코드는 수정하지 말고, 리팩토링 제안과 근거만 보고해라.

⸻

검토 원칙

* 실제 코드에서 확인되는 내용만 보고한다.
* 추측, 일반론, 취향 기반 조언은 제외한다.
* 문제가 없으면 해당 항목은 "문제 없음"으로 표시한다.
* 확신이 없으면 "확인 필요"로 표시하고, 왜 확인이 필요한지 짧게 적는다.
* 가능한 경우 파일:라인을 명시한다.
* 구조적 문제처럼 단일 라인으로 특정하기 어려운 경우에는 관련 파일 목록과 대표 위치를 제시한다.
* 테스트 실행, 타입 체크, 빌드 실행을 하지 않았다면 기능 정상 여부를 단정하지 않는다.
* 현재 잘 작동하고 안정적인 코드는 근거 없이 분리, 추상화, 재작성하라고 제안하지 않는다.
* 솔로 개발 단일 레포 프로젝트 기준으로 판단한다. 마이크로서비스 분리, 과도한 레이어링, 대규모 아키텍처 재설계는 제안하지 않는다.

⸻

1. 기능 안정성

검토할 것:

* 코드가 의도한 동작을 깨뜨릴 가능성이 있는가
* 에지 케이스 처리가 빠진 자리가 있는가
* 에러 핸들링이 누락되거나 과한 자리가 있는가
* Rust라면 unwrap, expect, Result 무시, panic 가능성이 부적절하게 남아 있는가
* TypeScript라면 any, as 남용, null/undefined 처리 누락, 비동기 에러 처리 누락이 있는가

주의:

* 실행하지 않고 확인할 수 없는 기능 문제는 "확인 필요"로 표시한다.
* 리팩토링과 무관한 기능 추가 제안은 하지 않는다.

⸻

2. 결합 강도

두 모듈/컴포넌트가 서로에 대해 얼마나 많이 알고 있는지 본다.
공유 지식이 적을수록 안전하다.

결합 강도는 다음 태그로 분류한다.

태그	의미	위험도
Intrusive	한 모듈이 다른 모듈의 내부 구현 디테일에 의존한다	매우 높음
Functional	비즈니스 로직이 여러 모듈에 중복되거나 분산되어 한쪽 변경이 다른 쪽 변경을 강제한다	높음
Model	데이터 모델/구조체를 여러 모듈이 직접 공유해서 모델 변경이 연쇄된다	중간
Contract	명확한 인터페이스, 타입, 이벤트, API 계약으로만 소통한다	낮음

검토할 것:

* Contract로 충분한데 Model 또는 Functional 결합이 된 곳이 있는가
* 내부 구현 세부사항을 외부 모듈이 알고 있는가
* 같은 비즈니스 규칙이 여러 곳에 흩어져 있는가
* 타입 공유가 편의 때문인지, 실제 도메인 계약 때문인지 구분한다

⸻

3. 경계와 거리

검토할 것:

* 모듈 경계가 책임에 맞게 그어져 있는가
* 강하게 결합된 코드끼리 너무 멀리 떨어져 있는가
* 서로 관계없는 코드가 한 파일이나 모듈에 뭉쳐 있는가
* import / dependency 방향이 한 방향으로 흐르는가
* 순환 의존 또는 사실상의 순환 의존이 있는가
* UI, I/O, 외부 API, 도메인 로직, 저장소 접근이 부적절하게 섞여 있는가

판단 기준:

* 강하게 결합된 것끼리는 가까이 있어도 된다.
* 멀리 떨어져야 할 것은 안정적인 계약으로만 연결되어야 한다.
* 단순히 파일이 크다는 이유만으로 분리 제안하지 말고, 변경 이유가 여러 개인지 확인한다.

⸻

4. 변동성 정렬

가장 중요한 검토 항목이다.

검토할 것:

* 자주 바뀌는 코드와 안정적인 코드가 분리되어 있는가
* 자주 바뀌는 컴포넌트가 안정적인 코어와 강하게 결합되어 있는가
* 외부 포맷 파서, UI, 설정, 실험적 로직, 외부 API 연동이 코어 로직을 흔들 가능성이 있는가
* 특정 부분이 바뀌면 변경이 어디까지 전파되는가
* 안 바뀔 레거시 코드나 외부 라이브러리 래퍼는 강한 결합이 있어도 실제 위험이 낮은지 판단한다

⸻

5. 책임 분리

검토할 것:

* 한 모듈/함수/구조체가 단일 책임을 지키는가
* 여러 이유로 바뀌는 모듈이 있는가
* 함수나 파일이 큰 경우, 책임 과다 때문인지 단순히 절차가 긴 것인지 구분한다
* 도메인 판단, UI 상태, 저장소 접근, 외부 호출, 포맷 변환이 한곳에 섞여 있는지 본다

주의:

* 크기 자체를 문제로 보지 말 것.
* 실제 변경 이유가 여러 개인 경우에만 책임 분리를 제안할 것.

⸻

6. 코드 품질

모듈 내부의 미시적 품질을 본다.

검토할 것:

* 네이밍이 의도를 드러내는가
* 매직 넘버 / 매직 문자열이 있는가
* 테스트하기 어려운 부수 효과가 섞여 있는가
* 의존성 주입 또는 mock이 필요한 위치에서 불가능한 구조인가
* 같은 로직이 반복되는가
* 단, 성급한 추상화보다 명시적 중복이 나은 경우는 문제로 보지 않는다
* 해당 언어의 관용적 패턴을 따르는가

⸻

검토 시 계속 유지할 질문

* 이 구조는 모듈 간 공유 지식을 늘리는가, 줄이는가?
* 이 결합은 실제로 자주 바뀌는 부분에 있는가?
* Contract로 충분한데 Model이나 Functional 결합을 하고 있지는 않은가?
* 리팩토링이 실제 변경 비용을 줄이는가, 아니면 보기 좋은 구조를 만들 뿐인가?
* 지금 필요한 추상화인가?
* 안정적으로 잘 작동하는 코드를 건드리는 비용이 이득보다 크지는 않은가?
