# Salon: 로컬 LLM 통합(v0.3) 진입 전 결정 - 멀티 페르소나 컨텍스트 조립 & KV 캐시 전략

## 목적

로컬 LLM이 실제 발화를 생성하는 단계로 들어가기 전에, 각 페르소나 호출에 **무엇을 어떤 순서로 주입할지(컨텍스트 조립)**와 **keep_alive / KV 캐시 전략**을 확정한다. 지금 당장 캐시를 구현하라는 게 아니라, 전략 합의와 인터페이스 설계가 목적이다.

## 배경

- Salon은 멀티 LLM 페르소나가 한 방에서 잡담하는 도구다. v0.1은 LLM 없이 가짜 발화로 리듬 엔진(Hawkes + gate + RRF)만 검증 완료했다. 다음 LLM 통합 단계에서 ollama(gemma4:e4b, 로컬 또는 cloud)로 실제 발화를 생성할 예정이다.
- 화자 선택(누가 언제 말하나)은 엔진이 결정하고, 발화 내용만 LLM이 생성한다.
- 그리고 v0.3 이후 페르소나에 장기기억(tunaSalon 내부 장기기억 엔진, 잠정 이름 friend engine)을 얹을 계획이다. seCall의 하이브리드 검색 로직을 참고 포팅하는 별도 컴포넌트이며, 지금 구현 대상은 아니다. 다만 회상(과거 대화에서 검색해온 내용)을 프롬프트에 끼워넣을 슬롯을 이번 인터페이스에 미리 비워둬야 나중에 구조를 갈아엎지 않는다.

## 우리가 정리한 사실 (검증해줘)

1. LLM은 stateless다. 매 호출이 독립이고 이전 호출을 기억하지 않는다. "한 방에 같이 있다"는 맥락은 공유 대화 로그를 매 호출에 재주입해서 만든다(AutoGen GroupChat broadcast 패턴). 엔진은 이미 발화 history를 들고 있다.
2. gemma4:e4b는 128k 컨텍스트다(26b는 256k). 잡담 규모에선 컨텍스트 한계를 사실상 안 만난다. 슬라이딩 윈도우나 과거 발화 검색은 한참 뒤 문제다.
3. 호출마다 컨텍스트가 리셋되는 건 정상이고 설계 전제다. "계속 불러놓고 쓴다"가 의미 있는 건 두 가지뿐이다. (a) keep_alive로 모델 가중치를 메모리에 상주, (b) KV 캐시로 이미 처리한 프리픽스를 재사용.
4. 멀티 페르소나의 캐시 문제: 페르소나마다 시스템 프롬프트가 다르다. 페르소나를 번갈아 호출하면 프롬프트 맨 앞이 매번 바뀌어 KV 캐시 프리픽스가 깨진다.
5. 가설적 완화책: 변하지 않는 공통 부분(공유 대화 로그)을 프롬프트 앞에, 페르소나별 지시(너는 X다, 다음을 말해라)를 뒤에 배치하면 프리픽스 재사용이 가능하다. 단 이건 동작 필수가 아니라 최적화다. 잡담 규모면 매번 전체를 재주입해도 작은 모델은 충분히 빠를 가능성이 높다.

## 확인해줄 것 (코드 + 외부 사실)

1. 현재 엔진의 history/이벤트 구조가 LLM 프롬프트로 어떻게 연결될지. utterance 모듈이 지금 가짜 발화를 어떻게 만들고, 거기에 실제 LLM 호출을 끼우려면 인터페이스가 어디서 갈라지는지.
2. ollama의 실제 KV 캐시/prefix 캐시 동작. 위 5번(공통 앞 + 페르소나 뒤)이 ollama에서 실제로 캐시 재사용으로 이어지는지, 아니면 ollama가 프롬프트를 그렇게 캐싱하지 않는지. 추측 말고 ollama 문서나 동작으로 확인해줘.
3. ollama cloud의 prompt caching 지원 여부와 과금 단위. 로컬 ollama와 cloud에서 keep_alive / KV 캐시 제어가 어떻게 다른지. cloud가 캐시를 안 하면 매 발화마다 누적 로그 전체가 과금되므로 비용에 직결된다.

## 물어보는 것 (결정 필요)

1. 컨텍스트 조립을 어떤 구조로 갈지: 일단 매번 전체 로그를 재주입하는 단순한 방식으로 시작하고 캐시는 측정 후 도입할지, 아니면 처음부터 캐시 친화 순서로 설계할지.
2. keep_alive, num_ctx 등 ollama 설정 권고값(로컬/cloud 각각).
3. 지금 결정해 인터페이스에 반영해둘 것과 나중으로 미뤄도 되는 것의 경계. 조기 최적화는 피하되, 나중에 캐시 최적화로 갈 때 구조를 갈아엎지 않으려면 이번 인터페이스를 어떻게 열어둬야 하는지.
4. 회상 주입 슬롯: 나중에 장기기억 엔진이 검색해온 회상을 프롬프트의 어느 위치에 끼울지. 공유 로그와 페르소나 지시 사이 어느 지점이 캐시 친화적이면서(공통 프리픽스를 깨지 않으면서) 모델이 자연스럽게 참고하는지. 지금은 빈 슬롯만 열어두면 되고, 실제 회상 생성은 다음 단계다.

## 우리 입장 (지켜줘)

이건 전략 합의가 목적이지 지금 캐시를 구현하라는 게 아니다. 동작 가능한 최소(매번 재주입)부터 가고, 캐시 최적화는 실제로 느린 게 측정된 다음에 한다는 게 기본 방침이다. 장기기억 엔진(friend engine)도 마찬가지로 이번에 구현하는 게 아니라, 그 회상이 들어올 슬롯을 인터페이스에 비워두는 데까지만이다. 과하게 최적화나 미래 기능 설계로 앞서가지 마라. 코드베이스 확인과 ollama 실제 동작 확인을 근거로, 임의 변경 없이 권고안과 미결정 사항을 정리해서 제시해줘. 무엇을 이번 단계에서 결정해 인터페이스에 남기고 무엇을 미룰지를 분명히 답해줘.

## 결정 (2026-06-02, 채택 + 정정 2건)

**채택 (비용 0, 나중 갈아엎기 방지)**: 컨텍스트 조립을 단일 `assemble(...)` 함수로 추출하고, 회상 슬롯(`recall: Option<&str>`, 지금은 None)을 비워두며, 섹션 순서를 고정한다.

**섹션 순서**: 페르소나 지시는 `system` 필드(맨 앞)에 유지 - 작은 모델 페르소나 유지 우선. user 프롬프트 = [공유 로그] → [회상 슬롯(빈)] → [지금 답하라 지시].

**정정 1 - num_ctx**: gemma 128k는 모델 한계일 뿐, ollama 기본 num_ctx는 작다(API 문서 예시 1024). 명시 설정 안 하면 로그가 잘린다. 잡담은 발화 수십 개로 기본값을 금방 넘으므로, v0.3에서 LLM 붙일 때 **num_ctx를 처음부터 8k~16k로 올려둔다**(잘림 디버깅보다 깔끔, 비용 작음). 앞선 "128k라 한계 안 만난다" 설명의 빠진 단서.

**정정 2 - 캐시 친화 배치**: ollama가 자동 prefix 캐시를 문서상 보장하지 않는다(공식 메커니즘은 deprecated된 `context` 필드). 따라서 "공통 앞 = 캐시 재사용"은 **측정 전 기대 금지**. 캐시 친화(페르소나 뒤)와 작은 모델 페르소나 유지(페르소나 앞)는 충돌하는데, 캐시 이득이 불확실하니 페르소나 유지(system 앞)를 택하는 게 합리적이다.

**미결 (외부, 본격 사용 전 필수 확인)**: ollama cloud의 prompt caching 지원 + 과금 단위 - ollama.com 가격 페이지에서 확인. 캐시 없으면 발화마다 누적 로그가 선형 과금되어 비용 직결.

**미룸**: 실제 KV/prefix 캐시 최적화(느림 측정 + ollama 캐싱 확인 후), friend engine 본체(회상 슬롯만 예약).
