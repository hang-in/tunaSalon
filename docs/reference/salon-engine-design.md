# Salon - 대화 흐름 엔진 설계

로컬 LLM 페르소나들을 한 방에 초대해 스몰토크하게 하는 TUI(Rust) 앱.
이 문서는 페르소나가 아니라 그 뒤의 대화 흐름 엔진을 설계한다. 페르소나/모델은 갈아끼우는 부품이고, 본체는 이 엔진이다.

---

## 0. 핵심 전제 - 왜 기존 멀티에이전트와 다른가

기존 멀티에이전트 오케스트레이션(AutoGen GroupChat 등)은 거의 다 과제 수렴형이다. 문제를 풀고, 답이 나오면 종료한다. 화자 선택도 "누가 답을 아는가, 역할이 맞는가"로 정해지고, 종료는 승인 키워드 같은 명시적 신호로 일어난다.

Salon은 과제도 종료 조건도 없는 스몰토크다. 목표는 "정답 수렴"이 아니라 "대화가 죽지 않고 자연스럽게 흐르다 식는 것"이다. 그래서 최적화 대상 자체가 다르고, 화자 선택을 LLM 판단이 아니라 시간/맥락 기반 알고리즘으로 돌린다.

설계의 목적은 "정답 동작"을 박는 것이 아니다. 손잡이가 서로 분리돼 튜닝 가능한 놀이판을 만들고, 미터로 관찰하며 "어떤 대화는 10턴, 어떤 건 3턴, 어떤 건 한마디 하고 즉사"가 창발하는 파라미터 영역을 찾아 들어가는 것이다.

## 1. 차용 알고리즘과 출처

| 알고리즘 | 역할 | 비고 |
|---|---|---|
| Multivariate Hawkes process (self-exciting point process) | 발화 강도(언제/얼마나 말하고 싶은가)의 수학적 정식화 | 실제 대화 시퀀스 fit 선례 존재(Masuda 외, arXiv 1205.5109). 폭주 방지 안정 조건이 파라미터로 정의됨 |
| Reciprocal Rank Fusion (RRF) | 화자 선택(맥락 신호 융합) | seCall에서 BM25+벡터 융합에 쓰던 그 구조. 점수 스케일 무관, 순위만 융합 |
| GroupChatManager 패턴 (AutoGen) | conductor 루프(선택→생성→브로드캐스트) 골격 | "단순 규칙으로 충분하면 LLM을 쓰지 말라"는 공식 권고를 그대로 채택 |
| FSM transition 제약 (AutoGen) | 허용/금지 발화 전이 | 같은 페르소나 연속 발화 금지 등 |
| 임베딩 분산(BGE-M3) | 수렴/발산 측정 | seCall 임베딩 자산 전용. 대화 온도계 |

핵심 통찰: 동구님이 직관으로 만든 "발화게이지"는 사실 Hawkes intensity function이다. 시간에 따라 차오르고(base rate), 발화가 발화를 부르고(self/mutual excitation), 시간이 지나면 식는다(exponential decay). ad-hoc 게이지를 Hawkes로 정식화하면 폭주/고착 안정성을 파라미터 조건으로 통제할 수 있다.

## 2. 발화 강도층 - Multivariate Hawkes

각 페르소나 p는 하나의 점과정이고, 발화 욕구 강도 λ_p(t)를 가진다.

```
λ_p(t) = μ_p  +  Σ_j  α_{pj} · Σ_{t_k < t}  κ(t − t_k)
```

- μ_p (base rate): 페르소나의 기본 발화 성향. MBTI 발화성향이 여기로 들어간다(ENTP 높음, ISFJ 낮음).
- α_{pj} (교차 자극 행렬): 페르소나 j의 발화가 페르소나 p의 강도를 얼마나 자극하는가. 이 행렬이 곧 성격 케미다. critic은 누가 주장을 던지면 자극받고(α 높음), poet은 감정 발화에 자극받는다. self-excitation(α_{pp})은 "한번 말 시작하면 잠깐 더 말하고 싶음".
- κ(·) (감쇠 커널): 보통 지수 감쇠 exp(−β·Δt). β가 식는 속도. 자극이 시간이 지나면 base rate로 돌아간다.

안정 조건: 교차 자극 행렬의 spectral radius(또는 단변량이면 β > α)를 1 미만으로 묶으면 process explosion이 수학적으로 차단된다. 동구님이 걱정한 "멈출 줄 모르는 수다 기계"의 해법이 여기 있다. 이 조건을 깨면 의도적으로 "발작 모드"도 만들 수 있다(테스트용).

bursty 특성: Hawkes는 이벤트가 몰렸다 뜸했다 하는 burst를 자연 생성한다. "10턴 / 3턴 / 즉사"의 분포가 별도 로직 없이 강도 동역학에서 창발한다.

## 3. 화자 선택층 - 게이트 + RRF

Hawkes는 "언제/얼마나"의 시간 강도를 준다. RRF는 "지금 누구를"의 맥락을 결정한다. 둘은 시간 축과 맥락 축으로 보완한다.

### 3-1. 침묵 게이트
틱마다 모든 λ_p(t)를 본다. 최대 강도가 임계 θ를 넘는 후보가 없으면 아무도 말하지 않고 방이 조용해진다. 침묵이 길어지면 base rate가 강도를 서서히 밀어올려 누군가 결국 θ를 넘고 먼저 말을 건다(idle 발화).

### 3-2. RRF 융합
θ를 넘은 후보들 사이에서 최종 1명을 RRF로 뽑는다.

```
score(p) = Σ_i  1 / (k + rank_i(p))
```

융합 신호(각 신호마다 후보를 순위 매김):
- Hawkes 강도 순위 (얼마나 말하고 싶은가)
- 관심도 (직전 화제가 내 역할 키워드에 걸리는 정도)
- 발화 균형 (최근 적게 말한 순 - 독점 방지)
- 잔향 (과거 화제 재소환 욕구)
- 난수 (예측 불가성, 가끔 엉뚱한 페르소나가 끼어듦)
- 사람 반응성 (사람 발화 직후 가중)

k 튜닝: k를 크게 잡으면 1등 쏠림이 완만해져 여러 페르소나가 돌아가며 말한다(잡담 리듬). k가 작으면 한 명이 독점한다(회의 리듬).

### 3-3. FSM 전이 제약
RRF 위에 금지 전이를 얹는다. 같은 페르소나 2연속 금지, critic 직후엔 friend 가중 등. 화자 선택 결과를 전이 행렬로 필터링한다.

## 4. 거시층 - 수렴/발산과 메타 컨트롤러

발화 강도층/화자 선택층은 "누가 언제"라는 미시 결정이다. 거시층은 "대화 전체가 계속될지 마무리될지"를 본다.

- 측정: 최근 N개 발화를 BGE-M3로 임베딩해 벡터 분산을 본다. 분산이 작아지면(발화들이 서로 비슷해짐) 수렴, 새 개념이 계속 나오면(분산 커짐) 발산. 키워드 중복도로 더 싸게 근사 가능.
- 메타 컨트롤러: 이 값으로 미시층 파라미터를 조정한다. 수렴이 임계를 넘으면(다 한 얘기) 모든 base rate μ_p를 낮춰 방을 식히고, 발산이 높으면 그대로 둔다. 이 컨트롤러는 LLM이 아니라 휴리스틱 레이어다. "메타"는 LLM이라는 뜻이 아니라 미시 위에 얹힌 조정 층위라는 뜻이다.

## 5. 사람 참여

사람은 Hawkes 구조에 외부 이벤트로 자연스럽게 들어온다.

- 사람 발화 강도는 상시 무한(언제든 말할 수 있음).
- 사람이 말하면 큰 mark를 가진 이벤트로 처리해 모든 페르소나 λ_p를 강하게 자극(관심 집중)하고, 페르소나 강도를 일부 리셋해 사람에게 주목하게 한다.
- 화제 갱신: 사람 발화가 한동안 관심도/수렴-발산 측정의 기준점이 된다(주제 선점).

결과적으로 페르소나끼리 떠들다가도 사람이 끼면 화제가 사람 쪽으로 쏠리고, 사람이 빠지면 다시 자기들끼리 burst가 일어난다.

## 6. 생성층과 백엔드 - e4b 동시 호출 고려

설계상 매 틱 발화는 보통 1명이므로 LLM 호출도 0~1회다(결정은 싼 휴리스틱, 생성만 비싼 LLM - AutoGen 권고와 일치). 그러나 두 경우에 동시 호출이 필요하다.

- 테스트/벤치: 같은 e4b급 모델 3개에 서로 다른 페르소나를 주입해 동시에 응답시켜, persona collapse(작은 모델이 페르소나를 구분 못 하고 비슷하게 수렴) 여부를 비교한다.
- burst 표현: idle 후 여러 페르소나가 거의 동시에 끼어드는 순간.

백엔드 설계:
- PersonaRuntime이 백엔드 풀(ollama local/cloud, lmstudio)을 추상화한다.
- 동시성은 max_concurrent 세마포어로 보호한다. 테스트 모드는 3 병렬 허용, 스몰토크 모드는 기본 1(burst 시 제한적 병렬).
- e4b 3개를 로컬에 동시 상주시키면 메모리 부담이 크므로(대략 모델당 수 GB), ollama의 동시 로딩/큐 설정과 타임아웃 폴백을 둔다. cloud 백엔드는 부하가 머신이 아니라 과금으로 가니 호출 빈도 캡을 둔다.

## 7. 컴포넌트와 토글

| 모듈 | 역할 | 층위 | 토글 |
|---|---|---|---|
| HawkesEngine | 발화 강도 λ_p(t) 계산 | 미시 강도 | 코어 |
| SignalCollector + RRF Fuser | 화자 선택 | 미시 선택 | 신호별 on/off |
| SilenceGate | 발화 여부 판정 | 미시 선택 | on/off (끄면 항상 발화) |
| FSMTransition | 금지/허용 전이 | 미시 선택 | on/off |
| FlowMeter | 수렴/발산 측정 | 거시 | on/off |
| MetaController | 거시→미시 파라미터 조정 | 메타(휴리스틱) | on/off (끄면 미시만) |
| PersonaRuntime | LLM 생성, 백엔드 풀 | 생성 | 코어 |
| HumanChannel | 사람 인터럽트 | 입력 | 코어 |
| DebugMeter | 강도 곡선·선택 이유 로그 | 횡단 | on/off |

토글의 전제: 깨끗하게 켰다 껐다 하려면 각 모듈이 독립적이고 부작용이 없어야 한다. 모듈끼리 상태를 공유하거나 한쪽이 다른 쪽을 직접 건드리면, 꺼도 잔향이 남아 측정이 오염된다. "토글 가능하게 만든다"가 곧 "결합 없이 설계한다"는 제약이다.

## 8. 틱 사이클

매 틱(예: 고정 간격 또는 발화 직후):

1. 강도 갱신 - 경과 시간으로 모든 λ_p(t) 갱신(base rate + 누적 자극 − 감쇠).
2. 거시 평가 - FlowMeter가 수렴/발산 산출, MetaController가 base rate 보정.
3. 게이트 - 최대 강도가 θ 미만이면 침묵하고 다음 틱.
4. 신호 수집 + RRF - θ 넘은 후보를 맥락 신호로 융합해 랭킹.
5. FSM 필터 - 금지 전이 제거 후 최종 화자 1명 확정.
6. 생성 - 그 페르소나만 LLM 호출(ollama/lmstudio).
7. 갱신 - 발화 이벤트를 Hawkes 히스토리에 기록(다른 페르소나 자극), 화제·잔향 갱신, 미터 갱신.

사람 입력은 어느 틱에든 인터럽트로 들어온다.

## 9. 디버깅 미터 (관찰성)

창발·비결정 시스템은 미터 없이는 디버깅이 불가능하다. 왜 갑자기 다 조용한지, 왜 한 명이 독점하는지를 숫자로 봐야 손잡이를 돌릴 수 있다. TUI라 터미널에 그리기 쉽다.

표시 항목:
- 페르소나별 λ_p(t) 강도 막대 (실시간)
- 교차 자극이 발동한 경로 (누구 발화가 누구를 자극했나)
- 수렴/발산 게이지
- 이번 틱 화자 선택 이유 (어느 RRF 신호가 1등을 만들었는지)
- 침묵/발화 카운트, 대화 길이(턴)

### 9-1. 출력 sink 분리 - TUI / headless

엔진 코어는 터미널에 직접 그리지 않는다. 매 틱 관찰 데이터(강도, 선택 이유, 침묵/발화 카운트, 발화 이벤트)를 ObservationSink Contract로 내보내고, 두 구현이 이를 받는다.

- TUI 렌더러: DebugMeter를 인터랙티브하게 그린다(사람 관찰용).
- headless writer: 같은 데이터를 결정적 NDJSON으로 stdout에 흘린다. 고정 seed면 실행마다 동일 출력이라 스모크 테스트로 자동 검증된다.

이 분리는 7절 토글 전제(모듈 독립 = Contract 결합)와 가드레일의 변동성 정렬(UI를 코어에서 차단)을 그대로 따른다. 코어가 sink 종류를 모르므로, 사람이 보든 테스트가 읽든 엔진 동역학은 동일하다. 비결정 시스템을 회귀 검증하려면 결정적 출력 경로가 반드시 필요하므로, headless는 부가 기능이 아니라 v0.1부터 코어와 함께 들어간다.

## 10. 안정성 리스크와 시작 전략

가장 큰 위험은 컴포넌트 수가 아니라 MetaController의 피드백 루프다. 거시 상태가 미시 파라미터를 바꾸고 그 결과가 다시 거시에 영향을 주므로, 잘못 잡으면 진동(말했다 조용했다 발작)이나 고착(영원 침묵/영원 수다)이 난다. 제어이론의 안정성 문제와 같다.

시작 순서(토글 전략):
1. HawkesEngine + SilenceGate + RRF만으로 방을 굴린다. 거시 피드백 없이 미시만. 안정 조건(spectral radius < 1)으로 폭주가 안 나는지 미터로 확인.
2. FSM 전이를 얹어 같은 페르소나 연속/어색한 전이를 정리.
3. FlowMeter를 켜되 MetaController는 관찰만(파라미터 조정 끔). 수렴/발산 숫자가 의미 있게 움직이는지 본다.
4. MetaController를 약하게 켜서 식히기 피드백을 넣는다. 진동이 나면 게인을 낮춘다.

한 번에 다 켜면 방이 이상하게 굴 때 어느 모듈이 범인인지 못 가린다.

## 11. 핵심 튜닝 손잡이

| 파라미터 | 효과 |
|---|---|
| μ_p (base rate) | 페르소나 기본 수다력. MBTI 성향 |
| α 행렬 | 성격 케미. 누가 누구를 자극하나 |
| β (감쇠) | 식는 속도. 크면 빨리 조용 |
| spectral radius | 폭주/안정 경계. 1 미만 유지 |
| θ (게이트 임계) | 침묵 빈도. 높으면 조용한 방 |
| RRF k | 1등 독점 vs 분산. 잡담/회의 리듬 |
| 수렴/발산 임계 | 마무리 시점 |
| MetaController 게인 | 식히기 강도. 너무 크면 진동 |

## 12. 한 줄 정의

페르소나는 부품이고, Salon의 실체는 Hawkes 강도(언제) + RRF(누가) + 게이트(말할까) + 임베딩 수렴/발산(계속할까)을 미터로 관찰하며 튜닝하는 대화 흐름 엔진이다. 발화를 설계하는 것보다 침묵을 설계하는 것이 어려운, 좀 거꾸로 된 프로젝트다.
