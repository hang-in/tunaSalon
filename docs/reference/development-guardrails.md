# Salon 개발 가드레일

원본은 `docs/prompts/refactoring-review-prompt.md` (리뷰용 프롬프트 템플릿) 입니다.
본 문서는 그 안에서 리팩토링이 아닌 일반 개발 진행 시점에도 그대로 적용 가능한 원칙들을 발췌, 정리한 SSOT 입니다.
신규 코드 작성, 설계 결정, plan 단계 검토, 구현 중 자기 점검에 사용합니다.

## 0. 적용 범위

- 솔로 개발, 단일 레포 기준.
- 마이크로서비스 분리, 과도한 레이어링, 대규모 아키텍처 재설계는 권장하지 않습니다.
- Architect 는 plan 단계에서, Developer 는 구현 중, Reviewer 는 검토 시 동일한 기준선으로 사용합니다.

## 1. 판단 원칙

- 실제 코드에서 확인되는 내용만 보고하거나 변경합니다.
- 추측, 일반론, 취향 기반 조언은 제외합니다.
- 확신이 없는 항목은 "확인 필요" 로 표시하고 이유를 짧게 적습니다.
- 빌드, 타입 체크, 테스트를 실행하지 않았다면 정상 동작을 단정하지 않습니다.
- 잘 작동하는 안정적인 코드는 근거 없이 분리, 추상화, 재작성하지 않습니다.
- 성급한 추상화보다 명시적 중복이 나은 경우에는 중복을 유지합니다.

## 2. 결합 강도 4단계

새 모듈을 설계할 때, 두 모듈이 서로 얼마나 알고 있는지를 다음 태그로 분류합니다. 공유 지식이 적을수록 안전합니다.

| 태그 | 의미 | 위험도 |
|------|------|--------|
| Intrusive | 한 모듈이 다른 모듈의 내부 구현 세부에 의존 | 매우 높음 |
| Functional | 같은 비즈니스 로직이 여러 모듈에 중복, 분산되어 한쪽 변경이 다른 쪽 변경을 강제 | 높음 |
| Model | 데이터 모델, 구조체를 직접 공유해서 모델 변경이 연쇄 | 중간 |
| Contract | 명확한 인터페이스, 타입, 이벤트, API 계약으로만 소통 | 낮음 |

개발 중 자문할 것:
- Contract 로 충분한데 Model 또는 Functional 결합을 만들고 있지는 않은지.
- 타입 공유가 편의 때문인지, 실제 도메인 계약 때문인지.
- 같은 비즈니스 규칙이 여러 곳에 흩어져 있지는 않은지.

## 3. 경계와 거리

- 강하게 결합된 코드끼리는 가까이 둡니다.
- 멀리 떨어져야 할 것은 안정적인 계약으로만 연결합니다.
- import / dependency 방향은 한 방향으로 흐릅니다. 순환 의존 또는 사실상 순환 의존은 금지합니다.
- UI, I/O, 외부 API, 도메인 로직, 저장소 접근을 한 곳에 섞지 않습니다.
- 단지 파일이 크다는 이유만으로 분리하지 않습니다. 변경 이유가 여러 개인지 확인합니다.

## 4. 변동성 정렬 (가장 중요한 항목)

- 자주 바뀌는 코드와 안정적인 코드를 분리합니다.
- 자주 바뀌는 컴포넌트가 안정적인 코어와 강하게 결합되지 않도록 합니다.
- 외부 포맷 파서, UI, 설정, 실험 로직, 외부 API 연동은 코어 로직을 흔들지 못하게 차단합니다.

새 코드를 추가할 때 항상 다음을 명시합니다.
- 이 파일이 바뀌면 같이 바뀔 가능성이 있는 파일은 무엇인가.
- 전파 원인은 무엇인가.
- 결합 유형은 무엇인가 (Intrusive / Functional / Model / Contract).

## 5. 책임 분리

- 한 모듈, 함수, 구조체는 단일 책임을 지킵니다.
- 크기 자체는 문제가 아닙니다. 변경 이유가 여러 개일 때만 분리합니다.
- 도메인 판단, UI 상태, 저장소 접근, 외부 호출, 포맷 변환을 한 곳에 섞지 않습니다.

## 6. 코드 품질 미시 항목

- 네이밍이 의도를 드러내는가.
- 매직 넘버, 매직 문자열이 있는가.
- 테스트하기 어려운 부수 효과가 섞여 있는가.
- 의존성 주입 또는 mock 이 필요한 위치에서 가능한 구조인가.
- 같은 로직이 반복되는가. 단, 성급한 추상화보다 명시적 중복이 나은 경우는 문제로 보지 않습니다.
- 언어의 관용적 패턴을 따르는가.
- Rust 의 경우: `unwrap`, `expect`, `panic` 가능성이 부적절하게 남아 있지 않은지. `Result` 무시 금지.
- TypeScript 의 경우: `any`, `as` 남용 금지. `null` / `undefined` 처리, 비동기 에러 처리 누락 점검.

## 7. 항상 자문할 질문

- 이 구조는 모듈 간 공유 지식을 늘리는가 줄이는가.
- 이 결합은 실제로 자주 바뀌는 부분에 있는가.
- Contract 로 충분한데 Model 이나 Functional 결합을 하고 있지는 않은가.
- 지금 필요한 추상화인가.
- 안정적으로 잘 작동하는 코드를 건드리는 비용이 이득보다 크지는 않은가.

## 8. Salon 컨텍스트 적용 메모

`salon-engine-design.md` 8절은 "토글 가능성 = 모듈 독립성" 으로 명시합니다. 이는 본 가드레일 2장 결합 강도 Contract 등급과 4장 변동성 정렬과 정확히 일치하는 제약입니다.
각 모듈 (HawkesEngine, SignalCollector, SilenceGate, FSMTransition, FlowMeter, MetaController, PersonaRuntime 등) 은 서로 Contract 로만 소통해야 토글 시 잔향 없이 미터 측정이 가능합니다.

또한 10절 시작 전략 (Hawkes 코어 -> FSM -> FlowMeter 관찰 -> MetaController 단계적 활성화) 자체가 변동성 정렬의 적용 예시입니다. 한 번에 다 켜지 않는 이유는 진동, 고착이 일어났을 때 범인 모듈을 가릴 수 있어야 하기 때문이며, 이는 본 가드레일과 동일 원칙입니다.
