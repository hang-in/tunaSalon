![determinism](https://img.shields.io/badge/output-deterministic-informational)

주제를 던지면 LLM 페르소나들이 토론하는 브라우저 앱입니다. 서로 기억하고, 반박하고, 닉네임을 부르며 논쟁합니다.

다만 진짜 주인공은 페르소나가 아닙니다. tunaSalon의 본체는 누가 언제 말하고, 언제 조용해지고, 어떤 분위기에서 대화가 달아오르는지를 만드는 **대화 흐름 엔진**입니다. 그 위에 *연출자(producer)* 레이어가 얹혀 주제를 어떤 종류의 토론으로 만들지 정합니다 - 법정, 정책 다툼, 가치 딜레마. 사용자는 언제든 1급 참여자로 끼어들 수 있고, 페르소나들이 곧장 반응합니다. (원래는 엔진만 보는 터미널 장난감으로 시작했고, 지금도 `cargo run`으로 라이브 미터를 그렇게 볼 수 있습니다.)

보통 멀티에이전트 시스템은 문제를 풀기 위해 돌아갑니다. 하지만 토론도, 잡담도 다릅니다. 누군가는 끼어들고, 누군가는 듣고만 있고, 대화는 잠깐 달아올랐다가 다시 식습니다.

> 이 README 대부분은 엔진을 한 겹씩(v0.1~v0.10) 따라갑니다. 그 엔진이 굴리는 제품은 [브라우저 토론 앱](#브라우저-토론-앱)을 참조하세요.

tunaSalon은 그 흐름을 만들기 위한 실험입니다. v0.6은 방이 아직 살아있는지, 같은 말을 맴돌고 있는지를 수치로 보여주는 대화 온도계를 붙입니다.

LLM에게 "다음은 누가 말할까?"를 묻지 않습니다. 화자 선택과 침묵은 가벼운 알고리즘이 담당하고, LLM은 실제 발언 내용만 생성합니다.

> 말하게 하는 것보다, 자연스럽게 조용해지게 만드는 쪽이 더 어렵습니다.
> tunaSalon은 그 이상한 문제를 다루는 프로젝트입니다.

## 보통 페르소나 채팅과 뭐가 다른가

일반적인 페르소나 채팅은 대개 단순합니다.

- 순서대로 대답한다
- 사용자가 부른 쪽이 답한다
- 모든 페르소나가 매번 한마디씩 한다

이 방식에는 대화의 리듬이 없습니다. 말할 타이밍도 없고, 끼어들기 감각도 없고, "이번엔 그냥 듣고 있겠다"는 선택도 없습니다.

tunaSalon은 각 페르소나마다 발언 게이지를 계산합니다.

발언 게이지는 시간이 지나면 조금씩 차오르고, 말을 하고 나면 내려갑니다. 누군가의 말에 자극받으면 다시 올라갑니다. 아무도 충분히 말할 상태가 아니면 방은 조용해집니다. 여러 명이 동시에 말할 수 있는 상태가 되면, 엔진이 그중 한 명만 고릅니다.

그래서 대화는 항상 이어지지 않습니다.

- 어떤 방은 쉴 새 없이 떠들고,
- 어떤 방은 몇 마디 하다 조용해지고,
- 어떤 방은 한 페르소나가 계속 끼어들고,
- 어떤 방은 조용하던 페르소나가 특정 분위기에서 갑자기 말하기 시작합니다.

성격은 대사에만 있는 게 아니라 타이밍에도 있습니다.

## 알고리즘을 아주 짧게 설명하면

tunaSalon의 엔진은 네 가지 손잡이로 움직입니다.

| 손잡이 | 의미 |
|---|---|
| `μ` 말수 | 페르소나의 기본 수다력 |
| `λ` 발언 압력 | 지금 이 페르소나가 얼마나 말할 차례에 가까운지 |
| `θ` 침묵 문턱 | 이 선을 못 넘으면 아무도 말하지 않음 |
| `RRF` 화자 선택 | 말할 후보가 여럿일 때 최종 화자를 고르는 방식 |

매 틱마다 엔진은 이렇게 움직입니다.

```
각 페르소나의 발언 압력을 갱신한다
→ 아무도 침묵 문턱을 넘지 못하면 조용히 있는다
→ 후보가 있으면 한 명을 고른다
→ 그 페르소나가 말한 것으로 기록한다
→ 발언 압력과 방 분위기를 다시 갱신한다
```

여기까지는 LLM이 필요 없습니다. 가짜 발언만으로도 "누가 언제 말하는지"의 리듬을 볼 수 있습니다.

## 손잡이를 돌리면 무엇이 달라지나

`cargo run -- --sweep`으로 같은 조건에서 침묵 문턱 `θ`를 바꿔보면 방 분위기가 달라집니다.

말수는 다음과 같이 고정했습니다.

```
friend     μ = 0.80
chaos      μ = 0.70
summarizer μ = 0.25
```

결과는 다음과 같습니다.

```
θ=0.40  침묵   0   friend 100  chaos 100  summarizer 0
θ=0.65  침묵 100   friend  62  chaos  38  summarizer 0
θ=0.78  침묵 171   friend  29  chaos   0  summarizer 0
```

문턱이 낮으면 방이 쉬지 않고 떠듭니다. 문턱이 적당하면 말과 침묵이 섞인 리듬이 생깁니다. 문턱이 높으면 가장 수다스러운 페르소나만 겨우 말합니다.

같은 페르소나라도, 침묵 문턱 하나로 방의 리듬이 완전히 달라집니다.

## 케미 - 누가 누구를 자극하는가

v0.2에서는 교차 자극 `α`가 들어갑니다.

한 페르소나의 발언이 다른 페르소나의 발언 압력을 끌어올립니다. 이 값이 곧 방의 케미입니다.

예를 들어 같은 페르소나 구성을 두고 방 분위기만 바꾸면 결과가 달라집니다.

```
preset=Calm      침묵 99   friend 67  chaos 34  summarizer  0
preset=Argument  침묵  0   friend 76  chaos 76  summarizer 48
```

Calm 방에서는 조용한 summarizer가 한 번도 말하지 않습니다. Argument 방에서는 교차 자극이 커져서 summarizer도 48번이나 끼어듭니다.

페르소나는 그대로인데, 방 분위기가 조용한 쪽을 대화로 끌어낸 것입니다.

## TUI 미터

tunaSalon은 대화 흐름을 숫자로 관찰합니다.

```
┌events──────────────────────────────────────┐┌gauges────────────────────┐
│t8 (silence)                                ││Chaos Guest               │
│t9 (silence)                                ││########|.... 0.63        │
│t10 Friendly Regular                        ││########|.... 0.67        │
│                                            ││Quiet Summarizer          │
│                                            ││###.....|.... 0.25        │
│                                            ││speak 11  silence 6       │
└────────────────────────────────────────────┘└──────────────────────────┘
```

막대 안의 `|`가 침묵 문턱입니다. 막대가 그 선을 넘은 페르소나만 말할 수 있습니다.

하단의 `reason`은 이번 틱에서 왜 그 페르소나가 선택됐는지를 보여줍니다.

- 발언 압력이 가장 높았는지
- 최근 덜 말해서 균형상 선택됐는지
- 무작위 신호가 작동했는지

창발 시스템은 미터 없이 디버깅할 수 없습니다. tunaSalon은 대화가 왜 조용해졌는지, 왜 특정 페르소나가 독점하는지 숫자로 보면서 튜닝합니다.

## 로컬 LLM 연결 (v0.3)

v0.3에서 Ollama를 붙여 페르소나가 실제 대사를 생성합니다. **누가 언제 말하는지**는 여전히 엔진이 결정적으로 정하고, LLM은 발언 내용만 채웁니다.

기본 실행은 LLM 없이 돌아갑니다(FakeBackend). v0.1 출력과 바이트 단위로 동일하고, 네트워크도 필요 없습니다. `--llm`을 붙이면 opt-in됩니다.

실제 텍스트가 생기면 내용 기반 RRF 신호 두 가지가 활성화됩니다. **관심도**(새 화제가 이 페르소나를 얼마나 끌어당기는지)와 **잔향**(방금 들은 말을 아직 처리 중인지). 이 두 신호는 기존 λ/균형/난수 신호 위에 쌓입니다.

`persona_collapse` 예제는 같은 모델에 서로 다른 페르소나 프롬프트를 주입해 출력을 나란히 기록합니다. 작은 모델이 시간이 지나도 페르소나를 유지하는지, 아니면 결국 같은 말투로 무너지는지를 관찰합니다.

## 동시 호출 + 혼합 모델 (v0.4)

v0.4에서 **백엔드 풀**을 도입합니다. 프로토콜은 두 가지입니다.

- **Ollama** `/api/generate` - 예: `gemma4:31b-cloud`(클라우드, 동시성 상한 3)
- **OpenAI 호환** `/v1/chat/completions` - 예: 지인서버 vLLM (`qwen3.6-35b-fast`, 동시성 상한 1)

**페르소나별 라우팅**으로 한 방 안에서 모델을 섞을 수 있습니다. 일부 페르소나는 한 백엔드, 나머지는 다른 백엔드로 보냅니다. `mixed_bench` 예제에서 둘을 같은 방에 넣습니다.

라이브 틱 루프는 **순차를 유지**합니다(1명/틱, 인과적 턴테이킹). 동시 호출은 `generate_batch`를 통한 비교/벤치 전용입니다. 같은 프롬프트를 여러 백엔드에 동시에 보내 페르소나 톤을 비교하거나 지연을 측정할 때 씁니다.

기본은 클라우드 모델입니다. 로컬 RAM/GPU 사용 없이, 로컬 데몬이 원격으로 요청을 프록시합니다. 로컬 모델 로딩은 막혀 있습니다.

백엔드별 세마포어가 동시 in-flight를 상한 안에 묶습니다. 4xx 응답이나 타임아웃이 오면 폴백 체인이 동작합니다(다음 백엔드, 또는 FakeBackend). 패닉 없습니다.

**실제 혼합 모델 출력** (`cargo run --example mixed_bench`):

```
cloud  : gemma4:31b-cloud (cap=3)   friend : qwen3.6-35b-fast (cap=1)
routing: summarizer → friend, others → cloud
opening> 오늘 비 와서 다들 약속 취소했대. 좀 심심하네.

[friend via cloud]      비 오는 날엔 원래 좀 늘어지기 쉽지. 여기 커피나 마시면서 멍 때려.
[chaos via cloud]       그럼 우리 집 거실에서 비 구경 대회나 열까?
[summarizer via friend] 혼자 남아 있는 공간은 생각할 시간이 충분해진다.
```

summarizer는 더 큰 지인서버 모델로 라우팅되어 더 조용하고 사색적인 톤이 나옵니다. 서로 다른 모델에 걸쳐서도 페르소나 톤 구분이 유지됩니다.

## 방에 참여하기 (v0.5)

v0.5는 tunaSalon이 처음부터 만들어온 것을 완성합니다. 사용자가 직접 방에 들어갈 수 있게 됐습니다.

`cargo run -- --chat`을 실행하면 세 패널로 구성된 TUI가 열립니다. 왼쪽에는 모든 발언이 페르소나별로 표시되는 **채팅 패널**, 오른쪽에는 각 페르소나의 λ 게이지와 침묵 문턱 θ를 실시간으로 보여주는 **게이지 사이드바**, 아래에는 사용자가 입력하는 **텍스트 입력창**이 있습니다.

메커니즘은 **HumanChannel**입니다. 사용자가 메시지를 보내면 Hawkes 엔진에 큰 외부 이벤트로 입력됩니다. 모든 페르소나의 발언 압력이 강하게 올라가고, 화제가 사용자 쪽으로 당겨집니다(설계 §5). 잡담 중이던 페르소나들이 사용자를 향해 돌아서고, 조용해지면 다시 자기들끼리의 리듬으로 돌아갑니다.

LLM 생성(~1.6초/줄)은 메인 스레드 밖에서 실행되므로 페르소나가 "생각하는" 동안에도 UI가 반응합니다. 답변은 한 번에 하나씩 생성됩니다. 인과적 턴테이킹은 그대로입니다. 페르소나마다 다른 백엔드로 라우팅할 수 있습니다(예: 클라우드 모델 + 지인서버 vLLM `qwen3.6-35b-fast`). 하나가 다운되면 폴백이 동작합니다.

`--chat`은 실제 터미널이 필요합니다. 비대화형 셸에서는 안내 메시지를 출력하고 종료합니다.

**라이브 데모** (`cargo run --example chat_demo` - 터미널 불필요, 비대화형으로 실행):

```
친구: Hey! What's up?
혼돈: I've decided to replace all your socks with slightly damp sponges. Ta-ta!
나: 안녕, 다들 비 와서 뭐해?          ← 사용자가 참여
친구: 난 그냥 집에서 뒹굴거리는 중! ㅋㅋㅋ   ← 페르소나가 사용자를 향해 반응 (한국어로 전환)
```

사용자가 말하면 방의 관심이 사용자 쪽으로 이동합니다.

## 대화 온도계 (v0.6 — FlowMeter)

방이 아직 달아오르고 있는지, 아니면 식어가고 있는지 어떻게 알 수 있을까요?

v0.6은 **수렴 게이지**를 추가합니다. 0에서 1 사이의 숫자로, 최근 발언들이 같은 단어를 반복하고 있는지(수렴 - 식어가는 방), 아니면 계속 새로운 화제를 던지고 있는지(발산 - 살아있는 방)를 나타냅니다.

구현은 의도적으로 가볍습니다. 최근 발언들의 **토큰 중복 근사치** - 평균 쌍별 Jaccard 유사도입니다. 모델이 필요 없습니다. 나중에 더 정밀한 신호가 필요해지면 BGE-M3 임베딩을 같은 인터페이스 뒤에 끼워 넣을 수 있습니다.

**관찰 전용**입니다. 게이지는 수치를 측정하고 보여줄 뿐, 엔진에 피드백을 주지 않습니다. 파라미터 변경도, 자동 조정도 없습니다. 그 피드백 루프는 v0.7(MetaController)입니다. 가장 불안정한 부분이라 의도적으로 마지막에 뒀습니다.

어디서 볼 수 있나요:
- TUI 사이드바: λ 게이지 옆에 실시간 수렴 게이지("흐름 수렴 …").
- `cargo run --example chat_demo`: 발언마다 `[흐름] 수렴` 수치 출력.

**실제 `chat_demo` 출력** - 혼돈 페르소나가 계속 새로운 화제를 던지기 때문에 수렴도는 거의 0에 머뭅니다.

```
혼돈: 사실 너의 발가락 사이에서 비밀 사회가 운영 중이라는 소문을 들었어...
  [흐름] 수렴 0.00
나: 얘들아 각자 자기소개 좀 해줄래?
  [흐름] 수렴 0.01
혼돈: 난 그냥 여기저기 불 지르고 다니는 혼돈의 전도사야!
  [흐름] 수렴 0.04
```

*(수렴 ∈ [0,1]: 0 = 완전 발산/새로운 화제/활발 - 1 = 반복/수렴/식어가는 방.)*

결정성 보장: 게이지는 발언 내용에서만 계산합니다. 기본 LLM-off 실행에서는 발언이 가짜이므로 게이지가 없고, 헤드리스 골든 출력은 바이트 단위로 동일하게 유지됩니다.

## 피드백 루프 완성 (v0.7 — MetaController)

v0.6은 수렴을 측정했습니다. v0.7은 **그것에 반응합니다**.

대화가 **수렴**하면 - 같은 말을 맴돌기 시작하면 - 엔진이 **μ(기본 말수)를 낮춥니다**. 냉각 계수 `mu_scale` (∈ [하한, 1])을 통해서입니다. 방은 말이 줄고, 침묵이 길어지고, 스스로 식어갑니다. 대화가 발산 중이고 활발하면 컨트롤러는 μ를 건드리지 않습니다.

이것은 **휴리스틱** 거시→미시 레이어입니다. LLM이 아닙니다. 한 방향만입니다: 수렴 → 냉각.

**안정성이 핵심 설계 과제입니다** - 피드백 루프는 진동하거나 고착될 수 있고, 이 부분이 시스템 전체에서 가장 불안정합니다. 두 가지 안전장치를 뒀습니다.
- **기본 게인은 약하게**: 냉각 반응은 의도적으로 느립니다(`SALON_META_GAIN`으로 조정 가능).
- **하한선**: `mu_scale`은 ~0.4 아래로 내려가지 않습니다. 방이 영구적으로 조용해지거나 고착되지 않습니다.

**관찰하기**: TUI 사이드바와 `chat_demo`에 "식힘 ×{값}" 줄이 표시됩니다. `1.00`은 냉각이 없는 상태입니다.

**콘텐츠 게이트 / 골든 보존**: LLM 콘텐츠가 없는 기본 결정적 실행에서는 수렴 신호가 없으므로 `mu_scale`이 1.0을 유지하고, 엔진은 이전과 바이트 단위로 동일합니다. 결정적 헤드리스 경로는 변하지 않습니다.

한 가지 덧붙이면: 혼돈방 데모에서 대화는 계속 발산하기 때문에 냉각 게이지가 `식힘 ×1.00`에 머뭅니다. 이것은 올바른 동작입니다 - 냉각은 대화가 실제로 한 화제를 맴돌 만큼 수렴 임계값을 넘어설 때만 작동합니다. 한 번도 수렴하지 않는 방은 냉각되지 않습니다.

## 장기기억 - friend engine 첫 증분 (v0.8)

원래 엔진 레이어 로드맵(v0.1~v0.7)은 완성됐습니다. v0.8은 별도 트랙을 엽니다: **기억하는 페르소나**. 같은 방에 있던 페르소나가 무슨 말이 오갔는지 회상해 다음 발화에 끼워 넣기 시작합니다.

첫 증분은 의도적으로 작게 잡았습니다.
- **참여 기반 기억**: 사건 `{방, ts, 화자, 내용}` + 각 방에 누가 있었는지를 인메모리로 저장. 페르소나는 **자기가 실제로 앉아 있던 방**만 회상합니다. 참여하지 않은 대화는 회상 금지.
- **키워드 회상**: 토큰 중복 점수(FlowMeter 토크나이저 재사용)로 현재 맥락에 맞는 과거 발화 상위 K개를 골라, v0.3이 이미 예약해둔 회상 슬롯에 주입.
- **회상 평가 하네스**: 진짜 이점. 헤드리스 시나리오에 정답(SSOT)과 distractor를 방마다 심어두고, 검색층을 재현율/정확도와 참여 격리로 자동 채점합니다 - 결정적으로.

아직 **없는 것**(이후): BGE-M3 의미 검색, SQLite 영속(L1), 망각, 캐릭터별 주관적 저장, 방을 가로지르는 인물 인상.

**콘텐츠 게이트 / 골든 보존**: 회상은 **라이브 채팅 경로에만** 연결됩니다. 결정적 헤드리스/driver 경로는 회상을 주입하지 않으므로, LLM 콘텐츠가 없으면 사건도 회상도 없고 출력은 v0.1과 바이트 단위로 동일합니다.

## 더 깊은 회상 + 영속 (v0.9)

v0.8은 인메모리 + 토큰 중복으로 기억했습니다. v0.9는 본인 레포 [seCall](https://github.com/hang-in/seCall)의 검색코어를 lift해 회상을 심화하고 영속화합니다 - 전부 `friend-engine` Cargo feature 뒤(기본 off → 기본 빌드는 추가 의존성 0, 골든 무손상):

- **한국어 형태소**(Stage 0): Lindera(ko-dic)로 토큰화해 조사/어미를 제거 - "비가 온다"가 지난 "비 온다"를 매칭(공백 분리는 놓침).
- **SQLite + FTS5 BM25**(Stage 1): 스토어가 실제 SQLite + FTS5 인덱스가 되고, 회상이 토큰 개수가 아니라 BM25(빈도/희소도/길이)로 랭킹(OR-match).
- **세션 넘는 기억**: `--chat`이 `~/.local/share/tunaSalon/memory.db`(`$SALON_MEMORY_DB` override)에 영속 → 페르소나가 재실행해도 기억. 테스트는 `:memory:`로 디스크 무접촉.

v0.9와 함께 `--chat` 방도 생동감↑ - 튜닝된 3-way config(교차자극 + 같은 화자 2연속 금지)와 `/topic`(스모크 `smoke_chat`로 게이트).

## 의미 회상 (v0.10)

v0.9 회상은 어휘 기반(형태소 위 BM25)이라 단어는 안 겹치는데 의미는 같은 발화를 놓칩니다. v0.10은 그 위에 **의미 회상**을 얹습니다 - `friend-engine-semantic` sub-feature 뒤(v0.9는 ML 스택 없이 그대로 빌드되고, 기본 빌드는 여전히 ML-free + 골든 무손상):

- **BGE-M3 임베딩**(Stage 2a): ONNX Runtime(`ort`, download-binaries, 선택 CoreML)으로 프로세스 안에서 임베딩 - 데몬 없음, ollama 아님. 이 맥 실측: 로드 ~3.8s, embed ~29ms, ~2.3GB. 결정적 `MockEmbedder`로 일상 테스트는 재현 가능.
- **HNSW ANN**(Stage 2b): 벡터를 usearch 인덱스(cosine)에, 각 임베딩은 사건 옆 SQLite BLOB으로 영속.
- **하이브리드 RRF**(Stage 2c): 회상이 BM25(어휘) leg와 벡터(의미) leg를 RRF(k=60)로 융합 - 양쪽 다 참여 격리.
- **라이브 배선**(Stage 2d): `--chat`이 모델이 있으면 실 BGE-M3 임베더를 로드(없으면 Mock 폴백). 임베더는 DB당 일관(`meta.embedder_kind`) - 바뀌면 전체 재임베딩으로 Mock/Ort 벡터가 섞이지 않게 합니다.

그래서 어휘는 다른데 의미는 같은 발화가 회상됩니다: 저장된 "강아지랑 산책 다녀왔어"가 쿼리 "반려동물 데리고 나갔어"에 떠오릅니다(실모델 `#[ignore]` 테스트로 검증).

## 브라우저 토론 앱

엔진 트랙(v0.1~v0.10)은 늘 *수단*이었습니다 - 생동감. 제품은 방이고, 그 방이 이제 브라우저에서 여는 **주제 기반 LLM 토론 앱**이 됐습니다(axum WebSocket + React, `--features "web redis-bus"`).

유능한 에세이만 주고받는 토론은 지루합니다. 재미는 **연출자(producer)** 레이어에서 나오고, 이건 갓파일을 안 키우려고 독립 모듈 `src/debate/`에 따로 삽니다.

- **DebatePlan** - 주제를 결정적으로(LLM 메타콜 없이) 토론 **모드**에 매핑합니다: `Courtroom`, `PolicyDuel`, `MoralDilemma`, `PersonalStakes`, `Forecasting`, `DesignReview`. "AI 판사가 공정할까?"는 법정형이, "AI 규제와 오픈소스"는 정책 다툼이 됩니다. 모드가 방을 붙잡아 주제에서 새지 않게 합니다.
- **형식 변주** - 매 턴이 같은 길이가 아니라, 턴마다 형식이 도립니다: 교차신문, 스틸맨 후 반박, 구체 사례 하나, 측정 가능한 임계값, 조건부 양보. 일부는 일부러 한두 문장이라 발언이 일률적으로 느껴지지 않습니다.
- **루프 차단 twist 카드** - 방이 수렴(FlowMeter)하거나 같은 화자가 반복하면, 연출자가 새 국면(오판 통계, 규제 부담으로 떠난 유지보수자)을 투입해 참가자가 반응하도록 강제합니다.

전부 생성 스냅샷에만 주입되고 저장 히스토리는 건드리지 않아, LLM-off 골든 경로는 바이트 동일하게 유지됩니다.

**제품으로서의 방:**

- **두 가지 방 생성** - "만들기"는 room id 기반 **결정적 랜덤 3명**으로(같은 주제 → 같은 셋), "직접 고르기"는 축(혈액형/MBTI/별자리/역할)으로 **2~3명을 직접 구성**하며 입장 전 닉네임을 미리 봅니다.
- **초대·내보내기** 토론 중 가능하며, 참가자는 가용 백엔드로 라우팅됩니다(클라우드 모델, 또는 지인 vLLM이 떠 있을 때 - 죽은 백엔드로는 보내지 않습니다).
- **읽기 좋은 채팅** - 메시지는 가벼운 마크다운으로 렌더되고, 상대를 부를 때 그 닉네임이 페르소나 색으로 강조됩니다.
- **로비 + 영속** - 방은 `rooms.db`에 체크포인트되어 참가자·주제·히스토리와 함께 복원됩니다.
- **발언 게이지** - 참가자별 Hawkes 욕구 λ가 θ 임계 선에 대한 막대로 표시됩니다. θ를 넘으면 발화 *자격*이 생기고(실제 한 명은 RRF가 선택), 방의 차오름-발화-억제-침묵 리듬이 그대로 보입니다.

**Redis 멀티세션**(`redis-bus` feature, `SALON_REDIS_URL`로 opt-in)은 *휘발성 코디네이션* 레이어입니다 - 방 command/event 스트림, owner lease, presence - 기억 저장소가 아닙니다. `LiveSession`은 방당 단일 writer로 남고 SQLite(`memory.db`, `rooms.db`)가 영속 SSOT입니다. Redis를 비워도 히스토리는 안 사라집니다.

이 제품 트랙 전체가 `web` / `redis-bus` feature 뒤에 있습니다. 기본 빌드와 headless 골든 출력은 무손상입니다.

## 실행하기

Rust만 있으면 됩니다. 기본 실행은 LLM도 네트워크도 필요 없습니다.

```bash
cargo run                                         # TUI 미터 보기. q 종료, space 일시정지
cargo run -- --chat                               # 방에 참여: 인터랙티브 채팅 (실제 터미널 필요)
cargo run --features "web redis-bus" -- --web --topic "AI 판사가 공정할까?"  # 브라우저 토론 앱 (주제 모드 + 방 빌더 + 초대)
cargo run -- --headless --ticks 200               # 틱당 한 줄 NDJSON 출력
cargo run -- --sweep                              # 파라미터 스윕
cargo run -- --room argument                      # 방 분위기 프리셋
cargo run -- --room chaos --fsm                   # 케미 + 같은 사람 2연속 금지
cargo run -- --theta 0.7 --k 5 --beta 0.4         # 손잡이 직접 조정
cargo run -- --llm                                # LLM opt-in (기본 클라우드 모델, 네트워크 필요)
cargo run --example persona_collapse              # 같은 모델, 두 페르소나 비교 (Ollama 필요)
cargo run --example mixed_bench                   # 클라우드 + 지인서버 vLLM 혼합 (두 백엔드 필요)
cargo run --example chat_demo                     # 흐름 수렴 수치 출력이 포함된 비대화형 채팅 루프
cargo run --features friend-engine -- --chat      # 세션 넘어 기억(형태소 + SQLite BM25 회상)
cargo run --features friend-engine-semantic -- --chat  # + 의미 회상(모델 있으면 BGE-M3 로드)
cargo test                                        # 291 tests (friend-engine면 297, web+redis-bus면 326)
```

주요 손잡이는 다음과 같습니다.

| 옵션 | 의미 |
|---|---|
| `--theta` | 침묵 문턱 |
| `--k` | RRF에서 1등 쏠림을 완화하는 값 |
| `--beta` | 발언 압력이 회복/감쇠되는 속도 |
| `--room` | 방 분위기 프리셋 |

같은 `--seed`를 쓰면 출력은 매번 같습니다. 그래서 헤드리스 모드로 자동 검증할 수 있습니다.

- **web - 주제 토론 앱:** 방을 주제 토론으로 만드는 브라우저 프런트엔드(axum WebSocket + React). 독립 연출자 모듈 `src/debate/`: 결정적 DebatePlan/모드 추론, 모드별 형식 변주, 루프 차단 twist 카드. 랜덤/직접 구성 방 생성, 초대·내보내기, 닉네임 강조 마크다운 채팅, 로비 영속(`rooms.db`), Redis 멀티세션 버스(휘발성 코디네이션 - SQLite가 SSOT). `web` / `redis-bus` feature 뒤 - 기본 빌드와 골든 무손상.

## 왜 만들었나

멀티에이전트 LLM 데모는 대부분 과제를 풀기 위해 돌아갑니다.

문제를 나누고, 역할을 주고, 답을 만들고, 종료합니다. 그건 회의나 작업에는 맞습니다.

하지만 잡담은 다릅니다.

잡담은 답을 향해 수렴하지 않습니다. 목표도 없고, 승인 키워드도 없고, 명확한 종료 조건도 없습니다. 그냥 누군가 말하고, 누군가는 듣고, 대화가 잠깐 달아올랐다가 식습니다.

tunaSalon은 그 흐름을 다루는 실험입니다.

대사를 잘 만드는 것보다, 말할 타이밍과 침묵을 만드는 쪽이 핵심입니다.
