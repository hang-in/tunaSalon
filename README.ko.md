# tunaSalon

![status](https://img.shields.io/badge/status-v0.5-blue)
![tests](https://img.shields.io/badge/tests-148%20passing-brightgreen)

![determinism](https://img.shields.io/badge/output-deterministic-informational)

사용자가 로컬 LLM 페르소나들과 잡담하는 터미널 채팅방 TUI 앱입니다.

다만 핵심은 페르소나 자체가 아닙니다. tunaSalon의 본체는 누가 언제 말하고, 언제 조용해지고, 어떤 분위기에서 대화가 달아오르는지를 만드는 대화 흐름 엔진입니다. v0.5에서 사용자가 그 방에 직접 참여하게 됐습니다. 한마디 치면 페르소나들이 사용자에게 반응하고, 조용히 있으면 다시 자기들끼리 잡담합니다.

보통 멀티에이전트 시스템은 문제를 풀기 위해 돌아갑니다. 하지만 잡담은 다릅니다. 정답도 없고, 명확한 종료 조건도 없습니다. 누군가는 끼어들고, 누군가는 듣고만 있고, 대화는 잠깐 달아올랐다가 다시 식습니다.

tunaSalon은 그 흐름을 만들기 위한 실험입니다.

LLM에게 "다음은 누가 말할까?"를 묻지 않습니다. 화자 선택과 침묵은 가벼운 알고리즘이 담당하고, LLM은 실제 발언 내용만 생성합니다.

> 말하게 하는 것보다, 자연스럽게 조용해지게 만드는 쪽이 더 어렵습니다.
> tunaSalon은 그 이상한 문제를 다루는 프로젝트입니다.

## 보통 페르소나 채팅과 뭐가 다른가

일반적인 페르소나 채팅은 대개 단순합니다.

- 순서대로 대답한다
- 사용자가 부른 쪽이 답한다
- 모든 페르소나가 매번 한마디씩 한다

이 방식에는 대화의 리듬이 없습니다. 말할 타이밍도 없고, 끼어들기 감각도 없고, "이번엔 그냥 듣고 있겠다"는 선택도 없습니다.

tunaSalon은 각 페르소나마다 발언 게이지를 계산합니다.

발언 게이지는 시간이 지나면 조금씩 차오르고, 말을 하고 나면 내려갑니다. 누군가의 말에 자극받으면 다시 올라갑니다. 아무도 충분히 말할 상태가 아니면 방은 조용해집니다. 여러 명이 동시에 말할 수 있는 상태가 되면, 엔진이 그중 한 명만 고릅니다.

그래서 대화는 항상 이어지지 않습니다.

- 어떤 방은 쉴 새 없이 떠들고,
- 어떤 방은 몇 마디 하다 조용해지고,
- 어떤 방은 한 페르소나가 계속 끼어들고,
- 어떤 방은 조용하던 페르소나가 특정 분위기에서 갑자기 말하기 시작합니다.

성격은 대사에만 있는 게 아니라 타이밍에도 있습니다.

## 알고리즘을 아주 짧게 설명하면

tunaSalon의 엔진은 네 가지 손잡이로 움직입니다.

| 손잡이 | 의미 |
|---|---|
| `μ` 말수 | 페르소나의 기본 수다력 |
| `λ` 발언 압력 | 지금 이 페르소나가 얼마나 말할 차례에 가까운지 |
| `θ` 침묵 문턱 | 이 선을 못 넘으면 아무도 말하지 않음 |
| `RRF` 화자 선택 | 말할 후보가 여럿일 때 최종 화자를 고르는 방식 |

매 틱마다 엔진은 이렇게 움직입니다.

```
각 페르소나의 발언 압력을 갱신한다
→ 아무도 침묵 문턱을 넘지 못하면 조용히 있는다
→ 후보가 있으면 한 명을 고른다
→ 그 페르소나가 말한 것으로 기록한다
→ 발언 압력과 방 분위기를 다시 갱신한다
```

여기까지는 LLM이 필요 없습니다. 가짜 발언만으로도 "누가 언제 말하는지"의 리듬을 볼 수 있습니다.

---

## 손잡이를 돌리면 무엇이 달라지나

`cargo run -- --sweep`으로 같은 조건에서 침묵 문턱 `θ`를 바꿔보면 방 분위기가 달라집니다.

말수는 다음과 같이 고정했습니다.

```
friend     μ = 0.80
chaos      μ = 0.70
summarizer μ = 0.25
```

결과는 다음과 같습니다.

```
θ=0.40  침묵   0   friend 100  chaos 100  summarizer 0
θ=0.65  침묵 100   friend  62  chaos  38  summarizer 0
θ=0.78  침묵 171   friend  29  chaos   0  summarizer 0
```

문턱이 낮으면 방이 쉬지 않고 떠듭니다. 문턱이 적당하면 말과 침묵이 섞인 리듬이 생깁니다. 문턱이 높으면 가장 수다스러운 페르소나만 겨우 말합니다.

같은 페르소나라도, 침묵 문턱 하나로 방의 리듬이 완전히 달라집니다.

## 케미 - 누가 누구를 자극하는가

v0.2에서는 교차 자극 `α`가 들어갑니다.

한 페르소나의 발언이 다른 페르소나의 발언 압력을 끌어올립니다. 이 값이 곧 방의 케미입니다.

예를 들어 같은 페르소나 구성을 두고 방 분위기만 바꾸면 결과가 달라집니다.

```
preset=Calm      침묵 99   friend 67  chaos 34  summarizer  0
preset=Argument  침묵  0   friend 76  chaos 76  summarizer 48
```

Calm 방에서는 조용한 summarizer가 한 번도 말하지 않습니다. Argument 방에서는 교차 자극이 커져서 summarizer도 48번이나 끼어듭니다.

페르소나는 그대로인데, 방 분위기가 조용한 쪽을 대화로 끌어낸 것입니다.

## TUI 미터

tunaSalon은 대화 흐름을 숫자로 관찰합니다.

```
┌events──────────────────────────────────────┐┌gauges────────────────────┐
│t8 (silence)                                ││Chaos Guest               │
│t9 (silence)                                ││########|.... 0.63        │
│t10 Friendly Regular                        ││########|.... 0.67        │
│                                            ││Quiet Summarizer          │
│                                            ││###.....|.... 0.25        │
│                                            ││speak 11  silence 6       │
└────────────────────────────────────────────┘└──────────────────────────┘
```

막대 안의 `|`가 침묵 문턱입니다. 막대가 그 선을 넘은 페르소나만 말할 수 있습니다.

하단의 `reason`은 이번 틱에서 왜 그 페르소나가 선택됐는지를 보여줍니다.

- 발언 압력이 가장 높았는지
- 최근 덜 말해서 균형상 선택됐는지
- 무작위 신호가 작동했는지

창발 시스템은 미터 없이 디버깅할 수 없습니다. tunaSalon은 대화가 왜 조용해졌는지, 왜 특정 페르소나가 독점하는지 숫자로 보면서 튜닝합니다.

## 로컬 LLM 연결 (v0.3)

v0.3에서 Ollama를 붙여 페르소나가 실제 대사를 생성합니다. **누가 언제 말하는지**는 여전히 엔진이 결정적으로 정하고, LLM은 발언 내용만 채웁니다.

기본 실행은 LLM 없이 돌아갑니다(FakeBackend). v0.1 출력과 바이트 단위로 동일하고, 네트워크도 필요 없습니다. `--llm`을 붙이면 opt-in됩니다.

실제 텍스트가 생기면 내용 기반 RRF 신호 두 가지가 활성화됩니다. **관심도**(새 화제가 이 페르소나를 얼마나 끌어당기는지)와 **잔향**(방금 들은 말을 아직 처리 중인지). 이 두 신호는 기존 λ/균형/난수 신호 위에 쌓입니다.

`persona_collapse` 예제는 같은 모델에 서로 다른 페르소나 프롬프트를 주입해 출력을 나란히 기록합니다. 작은 모델이 시간이 지나도 페르소나를 유지하는지, 아니면 결국 같은 말투로 무너지는지를 관찰합니다.

## 동시 호출 + 혼합 모델 (v0.4)

v0.4에서 **백엔드 풀**을 도입합니다. 프로토콜은 두 가지입니다.

- **Ollama** `/api/generate` - 예: `gemma4:31b-cloud`(클라우드, 동시성 상한 3)
- **OpenAI 호환** `/v1/chat/completions` - 예: 지인서버 vLLM (`qwen3.6-35b-fast`, 동시성 상한 1)

**페르소나별 라우팅**으로 한 방 안에서 모델을 섞을 수 있습니다. 일부 페르소나는 한 백엔드, 나머지는 다른 백엔드로 보냅니다. `mixed_bench` 예제에서 둘을 같은 방에 넣습니다.

라이브 틱 루프는 **순차를 유지**합니다(1명/틱, 인과적 턴테이킹). 동시 호출은 `generate_batch`를 통한 비교/벤치 전용입니다. 같은 프롬프트를 여러 백엔드에 동시에 보내 페르소나 톤을 비교하거나 지연을 측정할 때 씁니다.

기본은 클라우드 모델입니다. 로컬 RAM/GPU 사용 없이, 로컬 데몬이 원격으로 요청을 프록시합니다. 로컬 모델 로딩은 막혀 있습니다.

백엔드별 세마포어가 동시 in-flight를 상한 안에 묶습니다. 4xx 응답이나 타임아웃이 오면 폴백 체인이 동작합니다(다음 백엔드, 또는 FakeBackend). 패닉 없습니다.

**실제 혼합 모델 출력** (`cargo run --example mixed_bench`):

```
cloud  : gemma4:31b-cloud (cap=3)   friend : qwen3.6-35b-fast (cap=1)
routing: summarizer → friend, others → cloud
opening> 오늘 비 와서 다들 약속 취소했대. 좀 심심하네.

[friend via cloud]      비 오는 날엔 원래 좀 늘어지기 쉽지. 여기 커피나 마시면서 멍 때려.
[chaos via cloud]       그럼 우리 집 거실에서 비 구경 대회나 열까?
[summarizer via friend] 혼자 남아 있는 공간은 생각할 시간이 충분해진다.
```

summarizer는 더 큰 지인서버 모델로 라우팅되어 더 조용하고 사색적인 톤이 나옵니다. 서로 다른 모델에 걸쳐서도 페르소나 톤 구분이 유지됩니다.

## 방에 참여하기 (v0.5)

v0.5는 tunaSalon이 처음부터 만들어온 것을 완성합니다. 사용자가 직접 방에 들어갈 수 있게 됐습니다.

`cargo run -- --chat`을 실행하면 세 패널로 구성된 TUI가 열립니다. 왼쪽에는 모든 발언이 페르소나별로 표시되는 **채팅 패널**, 오른쪽에는 각 페르소나의 λ 게이지와 침묵 문턱 θ를 실시간으로 보여주는 **게이지 사이드바**, 아래에는 사용자가 입력하는 **텍스트 입력창**이 있습니다.

메커니즘은 **HumanChannel**입니다. 사용자가 메시지를 보내면 Hawkes 엔진에 큰 외부 이벤트로 입력됩니다. 모든 페르소나의 발언 압력이 강하게 올라가고, 화제가 사용자 쪽으로 당겨집니다(설계 §5). 잡담 중이던 페르소나들이 사용자를 향해 돌아서고, 조용해지면 다시 자기들끼리의 리듬으로 돌아갑니다.

LLM 생성(~1.6초/줄)은 메인 스레드 밖에서 실행되므로 페르소나가 "생각하는" 동안에도 UI가 반응합니다. 답변은 한 번에 하나씩 생성됩니다. 인과적 턴테이킹은 그대로입니다. 페르소나마다 다른 백엔드로 라우팅할 수 있습니다(예: 클라우드 모델 + 지인서버 vLLM `qwen3.6-35b-fast`). 하나가 다운되면 폴백이 동작합니다.

`--chat`은 실제 터미널이 필요합니다. 비대화형 셸에서는 안내 메시지를 출력하고 종료합니다.

**라이브 데모** (`cargo run --example chat_demo` - 터미널 불필요, 비대화형으로 실행):

```
친구: Hey! What's up?
혼돈: I've decided to replace all your socks with slightly damp sponges. Ta-ta!
나: 안녕, 다들 비 와서 뭐해?          ← 사용자가 참여
친구: 난 그냥 집에서 뒹굴거리는 중! ㅋㅋㅋ   ← 페르소나가 사용자를 향해 반응 (한국어로 전환)
```

사용자가 말하면 방의 관심이 사용자 쪽으로 이동합니다.

## 실행하기

Rust만 있으면 됩니다. 기본 실행은 LLM도 네트워크도 필요 없습니다.

```bash
cargo run                                         # TUI 미터 보기. q 종료, space 일시정지
cargo run -- --chat                               # 방에 참여: 인터랙티브 채팅 (실제 터미널 필요)
cargo run -- --headless --ticks 200               # 틱당 한 줄 NDJSON 출력
cargo run -- --sweep                              # 파라미터 스윕
cargo run -- --room argument                      # 방 분위기 프리셋
cargo run -- --room chaos --fsm                   # 케미 + 같은 사람 2연속 금지
cargo run -- --theta 0.7 --k 5 --beta 0.4         # 손잡이 직접 조정
cargo run -- --llm                                # LLM opt-in (기본 클라우드 모델, 네트워크 필요)
cargo run --example persona_collapse              # 같은 모델, 두 페르소나 비교 (Ollama 필요)
cargo run --example mixed_bench                   # 클라우드 + 지인서버 vLLM 혼합 (두 백엔드 필요)
cargo run --example chat_demo                     # 스크립트 사용자 턴이 포함된 비대화형 채팅 루프
cargo test                                        # 148 tests
```

주요 손잡이는 다음과 같습니다.

| 옵션 | 의미 |
|---|---|
| `--theta` | 침묵 문턱 |
| `--k` | RRF에서 1등 쏠림을 완화하는 값 |
| `--beta` | 발언 압력이 회복/감쇠되는 속도 |
| `--room` | 방 분위기 프리셋 |

같은 `--seed`를 쓰면 출력은 매번 같습니다. 그래서 헤드리스 모드로 자동 검증할 수 있습니다.

## 왜 만들었나

멀티에이전트 LLM 데모는 대부분 과제를 풀기 위해 돌아갑니다.

문제를 나누고, 역할을 주고, 답을 만들고, 종료합니다. 그건 회의나 작업에는 맞습니다.

하지만 잡담은 다릅니다.

잡담은 답을 향해 수렴하지 않습니다. 목표도 없고, 승인 키워드도 없고, 명확한 종료 조건도 없습니다. 그냥 누군가 말하고, 누군가는 듣고, 대화가 잠깐 달아올랐다가 식습니다.

tunaSalon은 그 흐름을 다루는 실험입니다.

대사를 잘 만드는 것보다, 말할 타이밍과 침묵을 만드는 쪽이 핵심입니다.
