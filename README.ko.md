![determinism](https://img.shields.io/badge/output-deterministic-informational)

주제를 던지면 LLM 페르소나들이 토론하는 브라우저 앱입니다. 서로 기억하고, 반박하고, 닉네임을 부르며 논쟁합니다.

다만 진짜 주인공은 페르소나가 아닙니다. tunaSalon의 본체는 누가 언제 말하고, 언제 조용해지고, 어떤 분위기에서 대화가 달아오르는지를 만드는 **대화 흐름 엔진**입니다. 그 위에 *연출자(producer)* 레이어가 얹혀 주제를 어떤 종류의 토론으로 만들지 정합니다 - 법정, 정책 다툼, 가치 딜레마. 사용자는 언제든 1급 참여자로 끼어들 수 있고, 페르소나들이 곧장 반응합니다. (원래는 엔진만 보는 터미널 장난감으로 시작했고, 지금도 `cargo run`으로 라이브 미터를 그렇게 볼 수 있습니다.)

보통 멀티에이전트 시스템은 문제를 풀기 위해 돌아갑니다. 하지만 토론도, 잡담도 다릅니다. 누군가는 끼어들고, 누군가는 듣고만 있고, 대화는 잠깐 달아올랐다가 다시 식습니다.

> 이 README 대부분은 엔진을 한 겹씩(v0.1~v0.10) 따라갑니다. 그 엔진이 굴리는 제품은 [브라우저 토론 앱](#브라우저-토론-앱)을 참조하세요.

tunaSalon은 그 흐름을 만들기 위한 실험입니다. v0.6은 방이 아직 살아있는지, 같은 말을 맴돌고 있는지를 수치로 보여주는 대화 온도계를 붙입니다.

LLM에게 "다음은 누가 말할까?"를 묻지 않습니다. 화자 선택과 침묵은 가벼운 알고리즘이 담당하고, LLM은 실제 발언 내용만 생성합니다.

> 말하게 하는 것보다, 자연스럽게 조용해지게 만드는 쪽이 더 어렵습니다.
> tunaSalon은 그 이상한 문제를 다루는 프로젝트입니다.

## 보통 페르소나 채팅과 뭐가 다른가

일반적인 페르소나 채팅은 대개 단순합니다.

- 순서대로 대답한다
- 사용자가 부른 쪽이 답한다
- 모든 페르소나가 매번 한마디씩 한다

이 방식에는 대화의 리듬이 없습니다. 말할 타이밍도 없고, 끼어들기 감각도 없고, "이번엔 그냥 듣고 있겠다"는 선택도 없습니다.

tunaSalon은 각 페르소나마다 발언 게이지를 계산합니다.

발언 게이지는 시간이 지나면 조금씩 차오르고, 말을 하고 나면 내려갑니다. 누군가의 말에 자극받으면 다시 올라갑니다. 아무도 충분히 말할 상태가 아니면 방은 조용해집니다. 여러 명이 동시에 말할 수 있는 상태가 되면, 엔진이 그중 한 명만 고릅니다.

그래서 대화는 항상 이어지지 않습니다.

- 어떤 방은 쉴 새 없이 떠들고,
- 어떤 방은 몇 마디 하다 조용해지고,
- 어떤 방은 한 페르소나가 계속 끼어들고,
- 어떤 방은 조용하던 페르소나가 특정 분위기에서 갑자기 말하기 시작합니다.

성격은 대사에만 있는 게 아니라 타이밍에도 있습니다.

## 알고리즘을 아주 짧게 설명하면

tunaSalon의 엔진은 네 가지 손잡이로 움직입니다.

| 손잡이 | 의미 |
|---|---|
| `μ` 말수 | 페르소나의 기본 수다력 |
| `λ` 발언 압력 | 지금 이 페르소나가 얼마나 말할 차례에 가까운지 |
| `θ` 침묵 문턱 | 이 선을 못 넘으면 아무도 말하지 않음 |
| `RRF` 화자 선택 | 말할 후보가 여럿일 때 최종 화자를 고르는 방식 |

매 틱마다 엔진은 이렇게 움직입니다.

```
각 페르소나의 발언 압력을 갱신한다
→ 아무도 침묵 문턱을 넘지 못하면 조용히 있는다
→ 후보가 있으면 한 명을 고른다
→ 그 페르소나가 말한 것으로 기록한다
→ 발언 압력과 방 분위기를 다시 갱신한다
```

여기까지는 LLM이 필요 없습니다. 가짜 발언만으로도 "누가 언제 말하는지"의 리듬을 볼 수 있습니다.

## 손잡이를 돌리면 무엇이 달라지나

`cargo run -- --sweep`으로 같은 조건에서 침묵 문턱 `θ`를 바꿔보면 방 분위기가 달라집니다.

말수는 다음과 같이 고정했습니다.

```
friend     μ = 0.80
chaos      μ = 0.70
summarizer μ = 0.25
```

결과는 다음과 같습니다.

```
θ=0.40  침묵   0   friend 100  chaos 100  summarizer 0
θ=0.65  침묵 100   friend  62  chaos  38  summarizer 0
θ=0.78  침묵 171   friend  29  chaos   0  summarizer 0
```

문턱이 낮으면 방이 쉬지 않고 떠듭니다. 문턱이 적당하면 말과 침묵이 섞인 리듬이 생깁니다. 문턱이 높으면 가장 수다스러운 페르소나만 겨우 말합니다.

같은 페르소나라도, 침묵 문턱 하나로 방의 리듬이 완전히 달라집니다.

## 케미 - 누가 누구를 자극하는가

v0.2에서는 교차 자극 `α`가 들어갑니다.

한 페르소나의 발언이 다른 페르소나의 발언 압력을 끌어올립니다. 이 값이 곧 방의 케미입니다.

예를 들어 같은 페르소나 구성을 두고 방 분위기만 바꾸면 결과가 달라집니다.

```
preset=Calm      침묵 99   friend 67  chaos 34  summarizer  0
preset=Argument  침묵  0   friend 76  chaos 76  summarizer 48
```

Calm 방에서는 조용한 summarizer가 한 번도 말하지 않습니다. Argument 방에서는 교차 자극이 커져서 summarizer도 48번이나 끼어듭니다.

페르소나는 그대로인데, 방 분위기가 조용한 쪽을 대화로 끌어낸 것입니다.

## TUI 미터

tunaSalon은 대화 흐름을 숫자로 관찰합니다.

```
┌events──────────────────────────────────────┐┌gauges────────────────────┐
│t8 (silence)                                ││Chaos Guest               │
│t9 (silence)                                ││########|.... 0.63        │
│t10 Friendly Regular                        ││########|.... 0.67        │
│                                            ││Quiet Summarizer          │
│                                            ││###.....|.... 0.25        │
│                                            ││speak 11  silence 6       │
└────────────────────────────────────────────┘└──────────────────────────┘
```

막대 안의 `|`가 침묵 문턱입니다. 막대가 그 선을 넘은 페르소나만 말할 수 있습니다.

하단의 `reason`은 이번 틱에서 왜 그 페르소나가 선택됐는지를 보여줍니다.

- 발언 압력이 가장 높았는지
- 최근 덜 말해서 균형상 선택됐는지
- 무작위 신호가 작동했는지

창발 시스템은 미터 없이 디버깅할 수 없습니다. tunaSalon은 대화가 왜 조용해졌는지, 왜 특정 페르소나가 독점하는지 숫자로 보면서 튜닝합니다.

## 브라우저 토론 앱

엔진 트랙(v0.1~v0.10)은 늘 *수단*이었습니다 - 생동감. 제품은 방이고, 그 방이 이제 브라우저에서 여는 **주제 기반 LLM 토론 앱**이 됐습니다(axum WebSocket + React, `--features "web redis-bus"`).

유능한 에세이만 주고받는 토론은 지루합니다. 재미는 **연출자(producer)** 레이어에서 나오고, 이건 갓파일을 안 키우려고 독립 모듈 `src/debate/`에 따로 삽니다.

- **DebatePlan** - 주제를 결정적으로(LLM 메타콜 없이) 토론 **모드**에 매핑합니다: `Courtroom`, `PolicyDuel`, `MoralDilemma`, `PersonalStakes`, `Forecasting`, `DesignReview`. "AI 판사가 공정할까?"는 법정형이, "AI 규제와 오픈소스"는 정책 다툼이 됩니다. 모드가 방을 붙잡아 주제에서 새지 않게 합니다.
- **형식 변주** - 매 턴이 같은 길이가 아니라, 턴마다 형식이 도립니다: 교차신문, 스틸맨 후 반박, 구체 사례 하나, 측정 가능한 임계값, 조건부 양보. 일부는 일부러 한두 문장이라 발언이 일률적으로 느껴지지 않습니다.
- **루프 차단 twist 카드** - 방이 수렴(FlowMeter)하거나 같은 화자가 반복하면, 연출자가 새 국면(오판 통계, 규제 부담으로 떠난 유지보수자)을 투입해 참가자가 반응하도록 강제합니다.

전부 생성 스냅샷에만 주입되고 저장 히스토리는 건드리지 않아, LLM-off 골든 경로는 바이트 동일하게 유지됩니다.

**제품으로서의 방:**

- **두 가지 방 생성** - "만들기"는 room id 기반 **결정적 랜덤 3명**으로(같은 주제 → 같은 셋), "직접 고르기"는 축(혈액형/MBTI/별자리/역할)으로 **2~3명을 직접 구성**하며 입장 전 닉네임을 미리 봅니다.
- **초대·내보내기** 토론 중 가능하며, 참가자는 가용 백엔드로 라우팅됩니다(클라우드 모델, 또는 지인 vLLM이 떠 있을 때 - 죽은 백엔드로는 보내지 않습니다).
- **읽기 좋은 채팅** - 메시지는 가벼운 마크다운으로 렌더되고, 상대를 부를 때 그 닉네임이 페르소나 색으로 강조됩니다.
- **로비 + 영속** - 방은 `rooms.db`에 체크포인트되어 참가자·주제·히스토리와 함께 복원됩니다.
- **발언 게이지** - 참가자별 Hawkes 욕구 λ가 θ 임계 선에 대한 막대로 표시됩니다. θ를 넘으면 발화 *자격*이 생기고(실제 한 명은 RRF가 선택), 방의 차오름-발화-억제-침묵 리듬이 그대로 보입니다.

**Redis 멀티세션**(`redis-bus` feature, `SALON_REDIS_URL`로 opt-in)은 *휘발성 코디네이션* 레이어입니다 - 방 command/event 스트림, owner lease, presence - 기억 저장소가 아닙니다. `LiveSession`은 방당 단일 writer로 남고 SQLite(`memory.db`, `rooms.db`)가 영속 SSOT입니다. Redis를 비워도 히스토리는 안 사라집니다.

이 제품 트랙 전체가 `web` / `redis-bus` feature 뒤에 있습니다. 기본 빌드와 headless 골든 출력은 무손상입니다.

## 실행하기

Rust만 있으면 됩니다. 기본 실행은 LLM도 네트워크도 필요 없습니다.

```bash
cargo run                                         # TUI 미터 보기. q 종료, space 일시정지
cargo run -- --chat                               # 방에 참여: 인터랙티브 채팅 (실제 터미널 필요)
cargo run --features "web redis-bus" -- --web --topic "AI 판사가 공정할까?"  # 브라우저 토론 앱 (주제 모드 + 방 빌더 + 초대)
cargo run -- --headless --ticks 200               # 틱당 한 줄 NDJSON 출력
cargo run -- --sweep                              # 파라미터 스윕
cargo run -- --room argument                      # 방 분위기 프리셋
cargo run -- --room chaos --fsm                   # 케미 + 같은 사람 2연속 금지
cargo run -- --theta 0.7 --k 5 --beta 0.4         # 손잡이 직접 조정
cargo run -- --llm                                # LLM opt-in (기본 클라우드 모델, 네트워크 필요)
cargo run --example persona_collapse              # 같은 모델, 두 페르소나 비교 (Ollama 필요)
cargo run --example mixed_bench                   # 클라우드 + 지인서버 vLLM 혼합 (두 백엔드 필요)
cargo run --example chat_demo                     # 흐름 수렴 수치 출력이 포함된 비대화형 채팅 루프
cargo run --features friend-engine -- --chat      # 세션 넘어 기억(형태소 + SQLite BM25 회상)
cargo run --features friend-engine-semantic -- --chat  # + 의미 회상(모델 있으면 BGE-M3 로드)
cargo test                                        # 291 tests (friend-engine면 297, web+redis-bus면 326)
```

주요 손잡이는 다음과 같습니다.

| 옵션 | 의미 |
|---|---|
| `--theta` | 침묵 문턱 |
| `--k` | RRF에서 1등 쏠림을 완화하는 값 |
| `--beta` | 발언 압력이 회복/감쇠되는 속도 |
| `--room` | 방 분위기 프리셋 |

같은 `--seed`를 쓰면 출력은 매번 같습니다. 그래서 헤드리스 모드로 자동 검증할 수 있습니다.

## 왜 만들었나

LLM 몇 개를 한 주제에 붙여놓으면 매 턴 전부 답합니다. 영원히. 끼어들지도, 참지도, 지루해하지도 않는 패널이죠. 논쟁은 그렇게 굴러가지 않습니다. 진짜 논쟁에는 *타이밍*이 있습니다. 누군가 끼어들고, 누군가는 입을 다물고, 방은 같은 지점을 맴돌다가 누군가 새 각도를 끌고 들어옵니다. 그 어느 것도 프롬프트에 없습니다. 리듬입니다.

그래서 tunaSalon은 리듬을 먼저 만듭니다. 누구의 발언 욕구가 차오르는지, 누가 방금 말했는지, 방이 언제 조용해져도 되는지. 그리고 거기서 캐릭터와 싸움이 떨어져 나오게 둡니다. 모델은 말만 채웁니다. 흥미로운 부분은 누가 그 말을 할지, 그리고 언제 침묵을 그대로 둘지를 정하는 쪽입니다.
